# HTTP Methods & Status Codes: The Language of the Web

When computers talk to each other on the web, they use a specific language. Just like you need verbs to communicate ("get," "create," "delete"), web APIs use **HTTP Methods**. And just like you need feedback ("done!", "error!"), APIs use **Status Codes**.

Let's learn this language together!

## The HTTP Methods: Your Action Verbs üé¨

Think of HTTP methods as commands you give to a librarian:

### 1. GET - "Please Show Me" üìñ

**What it does:** Retrieves information without changing anything.

**Real-world analogy:** 
- Looking at a book in the library (you don't take it home)
- Checking your bank balance (just viewing, not withdrawing)
- Reading a menu at a restaurant

**Example:**
```
GET /api/users/123
‚Üí "Show me information about user 123"
```

**Key rule:** GET should NEVER change data. It's read-only!

<ProgressCheckpoint section="get-method" xpReward={5} />

### 2. POST - "Please Add This" ‚ûï

**What it does:** Creates something new.

**Real-world analogy:**
- Submitting a new library card application
- Posting a new photo on Instagram
- Placing an order at a restaurant

**Example:**
```
POST /api/users
Body: { "name": "John", "email": "john@example.com" }
‚Üí "Create a new user with this information"
```

**Key rule:** POST creates new things. Each time you POST, you create another one!

<ProgressCheckpoint section="post-method" xpReward={5} />

### 3. PUT - "Replace Everything" üîÑ

**What it does:** Replaces an entire resource with new data.

**Real-world analogy:**
- Replacing your entire profile information
- Rewriting a document completely
- Replacing all ingredients in a recipe

**Example:**
```
PUT /api/users/123
Body: { "name": "John Doe", "email": "john@example.com", "age": 30 }
‚Üí "Replace ALL information for user 123 with this"
```

**Key rule:** You must send the COMPLETE object. Missing fields get deleted!

<ProgressCheckpoint section="put-method" xpReward={5} />

### 4. DELETE - "Remove This" üóëÔ∏è

**What it does:** Removes a resource.

**Real-world analogy:**
- Canceling your library card
- Deleting a photo from your phone
- Canceling an order

**Example:**
```
DELETE /api/users/123
‚Üí "Remove user 123 from the system"
```

**Key rule:** Once deleted, it's gone! (Usually)

## Quick Comparison Table üìä

| Method | Action | Example | Changes Data? |
|:-------|:-------|:--------|:--------------|
| **GET** | Read/View | "Show me user 123" | ‚ùå No |
| **POST** | Create | "Add a new user" | ‚úÖ Yes |
| **PUT** | Replace | "Replace user 123's info" | ‚úÖ Yes |

</InfoBox>

## Status Codes: The Server's Response üö¶

When you make a request, the server responds with a 3-digit number. Think of them like traffic lights telling you what happened!

### üü¢ 2xx Series: Success! (Green Light)

Everything worked perfectly!

**200 OK** - "Here's what you asked for!"
```
GET /api/users/123
‚Üê 200 OK
{ "id": 123, "name": "John" }
```

**201 Created** - "I successfully created the new thing!"
```
POST /api/users
‚Üê 201 Created
{ "id": 456, "name": "Jane" }
```

**204 No Content** - "Done! But there's nothing to show you."
```
DELETE /api/users/123
‚Üê 204 No Content
(No body - just success)
```

<ProgressCheckpoint section="2xx-codes" xpReward={5} />

### üü° 4xx Series: You Made a Mistake (Yellow Light)

These mean **YOU** (the client/browser/app) did something wrong.

**400 Bad Request** - "I don't understand what you're asking for."
```
POST /api/users
Body: { "name": "" }  ‚Üê Empty name!
‚Üê 400 Bad Request
{ "error": "Name is required" }
```

**401 Unauthorized** - "Who are you? Please log in first!"
```
GET /api/account/settings
(No login token provided)
‚Üê 401 Unauthorized
{ "error": "Please log in" }
```

**403 Forbidden** - "I know who you are, but you can't do that!"
```
DELETE /api/users/123
(Regular user trying to delete someone)
‚Üê 403 Forbidden
{ "error": "Only admins can delete users" }
```

**404 Not Found** - "I can't find what you're looking for."
```
GET /api/users/99999
‚Üê 404 Not Found
{ "error": "User not found" }
```

<InfoBox type="warning">
  **401 vs 403 - What's the difference?**
  - **401:** "I don't know who you are" (not logged in)
  - **403:** "I know who you are, but you're not allowed" (logged in but no permission)
</InfoBox>

<ProgressCheckpoint section="4xx-codes" xpReward={5} />

### üî¥ 5xx Series: Server Error (Red Light)

These mean the **SERVER** messed up. Not your fault!

**500 Internal Server Error** - "Something broke on our end. Sorry!"
```
GET /api/users/123
‚Üê 500 Internal Server Error
{ "error": "Database connection failed" }
```

**503 Service Unavailable** - "We're temporarily down. Try again later."
```
GET /api/users
‚Üê 503 Service Unavailable
{ "error": "Server is under maintenance" }
```

<InfoBox type="info">
  When you see a 5xx error, it's usually not something you can fix. The server team needs to investigate!
</InfoBox>

<ProgressCheckpoint section="5xx-codes" xpReward={5} />

## Status Code Cheat Sheet üìù

| Code | Name | Meaning | When to Use |
|:-----|:-----|:--------|:------------|
| **200** | OK | Success | GET, PUT requests |
| **201** | Created | New resource created | POST requests |
| **204** | No Content | Success, no data to return | DELETE, PUT requests |
| **400** | Bad Request | Invalid data sent | Validation errors |
| **401** | Unauthorized | Not logged in | Missing auth token |
| **403** | Forbidden | No permission | Insufficient privileges |
| **404** | Not Found | Resource doesn't exist | Invalid ID |
| **500** | Server Error | Server crashed | Bugs, database errors |

## Real-World Example: Creating a User üë§

Let's see how methods and status codes work together:

**Scenario 1: Success**
```

Body: { "name": "Alice", "email": "alice@example.com" }

‚Üê 201 Created
{
  "id": 789,
  "name": "Alice",
  "email": "alice@example.com"

```

**Scenario 2: Validation Error**
```

‚Üê 400 Bad Request
{
  "errors": {
    "name": "Name is required",
    "email": "Invalid email format"
  }
}
```

**Scenario 3: Not Logged In**
```

(No authentication token)

‚Üê 401 Unauthorized
{
  "error": "Please log in to create users"
}
```
